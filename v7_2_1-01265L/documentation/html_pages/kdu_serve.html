kdu_serve

Java: class Kdu_serve

[Declared in "../apps/kdu_server/kdu_serve.h"]

Synopsis

Provides the platform independent component of a powerful and flexible JPIP server system. This object provides support for cache modeling and the ability to handle multiple window contexts concurrently. Within each window context, new window requests are posed by calls to set_window, data is generated by calls to generate_increments, and the current state of the window request may be queried by calls to get_window. The object also provides support for the definition of per-context preferences, via calls to set_window_prefs, which may affect the way in which responses are generated.

From Kakadu version 6.5 on, the kdu_serve object commits to maintaining the state set up in each window context by set_window, until such time as window_finished is called (if at all), even if intervening calls to generate_increments dispatch all content relevant to the window of interest. That is, a window of interest is kept open even after all data has been generated by calls to generate_increments. This is done because more data might need to be generated in the event that a call to release_chunks returns some abandoned data chunks. By adopting this policy, it is possible for a server application to defer its processing of data chunk abandonment requests from a client until it can be sure that recent transmissions (or retransmissions) of those chunks will not arrive. In practice, there is no need to call window_finished at all, but it can provide a valuable way to release temporary resources for use within other window contexts.

The object updates its internal cache model (which spans all window contexts), whenever data is generated by calls to generate_increments. However, the object is also fully capable of processing all JPIP cache model manipulation statements in the most efficient way, delaying the cache model updates until convenient while ensuring synchronization of the cache model. The object is also fully capable of handling lost data, adjusting its cache model in accordance with information provided by the application about lost data chunks.

Many of these features are new to Kakadu version 6.4. Moreover, from version 6.4, the object is able to handle very large windows of interest, spanning any spatial region, across any number of codestreams, sequencing content in a manner which at least partially preserves the progressive quality experience while minimizing access to resources. The logic for this is quite intricate and extensive, yet invisible to the application. The object manages cache models, interchange codestreams and open codestream interfaces into the ultimate kdu_serve_target through a multi-level caching strategy which keeps only a very small working set of codesteams active at any given time and can potentially discard portions of its cache model if it becomes unwieldly.

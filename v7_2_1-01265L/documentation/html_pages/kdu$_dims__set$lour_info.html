kdu_image_dims::set_colour_info

void set_colour_info( int num_colours, bool has_premultiplied_alpha=false, bool has_unassociated_alpha=false, int colour_space_confidence=0, jp2_colour_space colour_space=JP2_iccLUM_SPACE, const kdu_byte * profile=NULL)

[Declared in "../apps/image/kdu_image.h"]

Go to class description.

Synopsis

This function may be used to specify some knowledge that you have of the colour space and/or presence of alpha components.

Arguments

num_colours [int]

This is the only non-default argument. You must at least know the number of colour channels and this value must, of course, be compatible with the number of image components that you have.

If this value is 0, you are declaring that you know nothing about the colour or alpha properties of the component samples. You may use the function with a 0 argument in this way to remove any previously declared knowledge of the colour representation. If profile is non-NULL, the number of colours can be retrieved using the profile.

has_premultiplied_alpha [bool]

Set this to true if you know that the image component which immediately follows the last colour component holds premultiplied alpha samples.

has_unassociated_alpha [bool]

Set this to true if you know that the image component which immediately follows the last colour component holds alpha samples which have not been premultiplied by the colour intensity samples. Of course, you cannot set both this argument and has_premultiplied_alpha to true.

colour_space_confidence [int]

If 0 or less, you have no idea what the actual colour space is and the colour_space argument will be ignored. If 2 or more, you know the JP2/JPX name of the colour space, which is given by the colour_space argument. A value of 1 means that you have a reasonable idea of the colour space, but cannot be certain that it precisely matches the value supplied by the colour_space argument.

colour_space [jp2_colour_space]

profile [const kdu_byte *]

If non-NULL, the colour_space argument is ignored. In this case, profile must point to a buffer containing a valid ICC profile, whose contents are copied internally. The profile's first 4 bytes provide a big-endian representation of the length of the profile (as is true for all ICC profiles) and the buffer must be at least this long.

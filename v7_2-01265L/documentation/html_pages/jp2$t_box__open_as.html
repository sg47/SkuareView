Synopsis

This function is similar to the first form of open, except that it does not usually read a box header.

If the ultimate_src object is a dynamic cache, this function actually invokes open directly, with box_locator and ultimate_src as arguments, after which it modifies the box type to the supplied box_type, limits the contents length to at most the value supplied by the contents_length argument, and completely ignores the contents_locator argument.

Otherwise, the ultimate_src object refers to a file, in which case the function does not actually read anything, but behaves as if a box were opened at the location identified by box_locator, recovering the supplied box_type and contents_length, whereupon the actual box contents were found to commence at the location identified by contents_locator. In this case, the function will not return false unless box_type is supplied as 0.

The principle application for this function is where a box header has already been read, the box has been closed, but it is later desired to open the box again. In that case, the use of this function can save the cost of file seeking (to go back and read the header again) and indeed can allow a box to be closed and re-opened even in cases where the underlying file or data source does not support seeking at all (so long as there are no intervening read operations of course). The jpb_source object uses this function to ensure that broadcast streams which are consumed sequentially do not require support for file seeking.

Of course, the function may have other applications, such as the typecasting of byte ranges from within otherwise opaque boxes such as jp2_free_4cc boxes or jp2_media_4cc boxes.
